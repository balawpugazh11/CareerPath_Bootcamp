# Master the Code: Key Snippets Explained

To speak confidently about this build, you should understand how the most important pieces of code actually work line-by-line. Here are three critical snippets explained in plain English.

---

## 1. How Passwords are Secured (Backend)

**File:** `src/controllers/authController.js` (Inside the `register` function)

```javascript
// 1. Generate a salt (a random string added to passwords to make them unique)
const salt = await bcrypt.genSalt(10);

// 2. Combine the salt with the plain text password and scramble it heavily
const password_hash = await bcrypt.hash(password, salt);

// 3. Save the scrambled hash to the database, NOT the real password
const user = await User.create({ name, email, password_hash, role: 'student' });
```

**Why you built it this way:** 
"We never store plain text passwords. If the database is compromised, hackers will only see scrambled gibberish. I chose `bcrypt` because it includes a 'salt workload' component (the `10`), making it incredibly slow for modern computers to brute-force or guess the encryption."

## 2. How the Server Verifies a User is Logged In (Backend)

**File:** `src/middleware/authMiddleware.js` (Inside `exports.protect`)

```javascript
// 1. Grab the token string from the HTTP Request header 
// (It looks like "Bearer eyJhbGciOiJIUzI1NiIsIn...")
token = req.headers.authorization.split(' ')[1];

// 2. Use the exact same SECRET KEY used during login to verify the token hasn't been tampered with
const decoded = jwt.verify(token, process.env.JWT_SECRET);

// 3. Find the user in the database based on the ID hidden inside the token payload
req.user = await User.findById(decoded.id);

// 4. Proceed to the next step of the request
next();
```

**Why you built it this way:**
"REST APIs are 'stateless', meaning the server instantly forgets who you are after every click. To fix this, I implemented JWTs (JSON Web Tokens). The client stores the token and sends it with every request. My `protect` middleware acts as a bouncer, mathematically proving the token was generated by my server before allowing access to private data."

## 3. How the Frontend Hides Pages from Logged Out Users

**File:** `src/components/auth/ProtectedRoute.jsx`

```javascript
const ProtectedRoute = ({ roles }) => {
  // 1. Try to read the token from the browser's persistent storage
  const token = localStorage.getItem('token');
  const user = JSON.parse(localStorage.getItem('user'));

  // 2. If it's missing, boot them back to the login screen immediately
  if (!token || !user) {
    return <Navigate to="/login" replace />;
  }

  // 3. If they are logged in, but their role isn't allowed here, boot them to their dashboard
  if (roles && roles.length > 0 && !roles.includes(user.role)) {
    return <Navigate to="/dashboard" replace />;
  }

  // 4. Everything looks good. Render whatever page they originally asked for!
  return <Outlet />; 
};
```

**Why you built it this way:**
"I built a custom Higher Order Component (HOC) called `ProtectedRoute` leveraging React Router's `<Outlet />` API. Instead of writing authentication logic on every single page, I wrap my sensitive routes in `App.jsx` with this component. It inspects `localStorage` instantly; if you don't have a token, it acts as a firewall and redirects you before the sensitive UI even has a chance to render."
